# MEMORY.md

Prescriptive workflow rules for this repository (complements `CLAUDE.md`).

**At-a-glance (standard workflow before stop‚Äîsame as the hook‚Äôs ‚Äútasks need your attention‚Äù list):** after a trunk push, **(1)** wait until CI shows **`status: completed`**, then **read** **`conclusion`** and job JSON; **(2)** if the work is already done, **TaskUpdate** / `swiz tasks complete` every open session task; **(3)** if not, finish the work before stop. **`task #unkn-1` does not waive (1)‚Äì(3).** Details below.

## Git and session closure

- **DO**: After finishing a coding task, stage all intended paths (`git add` on modified and new files), run `bun run typecheck` and `bun run lint` (or the scoped tests you touched), then commit with Conventional Commits (`<type>(<scope>): <summary>`, summary under ~50 characters; types: `feat`, `fix`, `refactor`, `docs`, `style`, `test`, `chore`).
- **DO**: Before treating the session as done or answering ‚Äúfinished‚Äù, run `git status` and ensure a clean working tree on `main` (or your intended branch)‚Äîno unstaged edits and no untracked directories left from the task.
- **DO**: Push to `origin main` after commit when work should land remotely: `git log origin/main..HEAD --oneline`, then `git push origin main` (use `swiz push-wait` / project push flow when cooldown or hooks apply).
- **DO**: Treat **CI verification and native-task closure as part of the same workflow block as push**, not only when a stop hook lists ‚Äúwait for CI‚Äù / ‚Äúmark tasks completed‚Äù‚Äîthis is the **proactive** version of the same **three** obligations the hook states: **(1) wait for CI to complete, then check results** for the pushed commit; **(2) if the work is already done**, **TaskUpdate** / `swiz tasks complete` each current-session task; **(3) if the work is still needed**, complete it before treating the session as done. Commands: right after `git push origin main`, run `SHA=$(git rev-parse HEAD)`, `gh run list --repo mherod/swiz --commit "$SHA" --limit 1 --json databaseId,status,conclusion`, then `gh run view <databaseId> --repo mherod/swiz --json conclusion,status,jobs` (and `gh run view <id> --log-failed` on non-success). Use evidence prefixes **`commit:`**, **`test:`**, **`file:`**, or **`note:`** only on `swiz tasks complete` ([swiz#300](https://github.com/mherod/swiz/issues/300)).
- **DON'T**: Report the session as finished, hand off, or invoke **stop** immediately after a successful push without at least one CI poll for that `SHA` plus task reconciliation‚Äîsame bar as the stop-hook ‚Äútasks needing attention‚Äù triad, **including when the hook shows `task #unkn-1`** (unstable id does not waive CI or tasks).
- **DON'T**: Stop or declare the task complete while `git status` still shows modified or untracked files from that task‚Äîstop hooks and collaboration checks expect committed (and usually pushed) work.
- **DO**: When stop (or another hook) surfaces **uncommitted changes** or an **ACTION REQUIRED** to commit/push, treat that as a hard gate: run `git status`, stage every path from the task (`git add` including new files under `src/dispatch/` or elsewhere), run `bun run typecheck` and scoped or full `bun test`, commit with Conventional Commits, run `git log origin/main..HEAD --oneline`, push with `git push origin main` (or `swiz push-wait` / `/push` flow when the repo uses it), then resume‚Äîhooks encode trunk hygiene; ignoring the signal leaves the tree dirty and blocks a clean stop.

## Stop hook: CI + native tasks (hard gate)

When stop reports **tasks needing attention** (e.g. ‚Äúwait for CI‚Äù, ‚Äúmark tasks completed‚Äù), run this sequence before attempting stop again‚Äî**DON'T** dismiss the block or repeat stop unchanged.

- **DON'T**: Ignore the banner **‚ÄúThere are tasks that need your attention before we can finish the session‚Äù**‚Äîit is the same contract as steps **1‚Äì3** below: wait for CI and **check results**, **TaskUpdate** / `swiz tasks complete` when work is already done, otherwise **complete remaining work** before stop.
- **DO**: Treat the stop-hook‚Äôs default triad as the minimum bar: **(1)** wait for CI on the relevant commit and **read** `conclusion` / job JSON (not only `gh run watch`), **(2)** if implementation is done, **TaskUpdate** / `swiz tasks complete` every open session task with valid evidence, **(3)** if not done, ship or task the remaining work‚Äîthen retry stop.
- **DON'T**: Skip the checklist because a line shows **`task #unkn-1`** (or similar)‚Äîthat usually means the hook could not bind a stable task id; CI + task hygiene still apply.
- **DO**: Map the hook‚Äôs literal lines to this section: **‚ÄúWait for CI to complete, then check results‚Äù** ‚Üí steps **1‚Äì2**; **‚Äúuse TaskUpdate to mark each current-session task as completed‚Äù** ‚Üí step **3** when work is already done; **‚ÄúIf the work is still needed, complete it before stopping‚Äù** ‚Üí step **3**‚Äôs **in_progress** / **pending** path until shipped.

1. **CI for your SHA** ‚Äî After push: `git fetch origin main`, `SHA=$(git rev-parse origin/main)`, `gh run list --repo mherod/swiz --commit "$SHA" --limit 1 --json databaseId,status,conclusion`, then `gh run watch <databaseId> --repo mherod/swiz --exit-status` (or `swiz ci-wait $SHA --timeout 600`). If the run is **`cancelled`** on a long Test step, check `.github/workflows/ci.yml` `timeout-minutes` on the `ci` job versus full `bun test --concurrent` duration. **‚ÄúWait for CI to complete, then check results‚Äù** means: poll `gh run view <id> --json status,conclusion` until **`status`** is **`completed`** (not only `in_progress`), then read **`conclusion`** and full job JSON with `gh run view <id> --json conclusion,status,jobs`‚Äî**DON'T** treat **`gh run watch`** alone as ‚Äúchecked results‚Äù. On non-success, use **`--log-failed`** (step 2).
2. **Read results** ‚Äî On failure: `gh run view <id> --log-failed`; fix, commit, push, and re-check CI for the new `HEAD`.
3. **Native tasks** ‚Äî If work is already shipped, use **TaskUpdate** / `swiz tasks complete` so every current-session task is **`completed`** with accepted evidence (`commit:`, `test:`, `file:`, `note:`‚Äîonly those prefixes are accepted by `swiz tasks complete`; use `note:` for CI-green confirmation if no test artifact applies). If work remains, keep or create **`in_progress`** / **`pending`** tasks and finish the implementation before stop.

- **DON'T**: Treat **‚Äúpushed‚Äù** or **‚Äúpre-push passed locally‚Äù** as session-complete while `origin/main`‚Äôs latest run for that commit is missing, `in_progress`, **`cancelled`**, or **`failure`**.
